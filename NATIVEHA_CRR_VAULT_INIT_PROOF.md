NativeHA queue manager with Cross-Region Replication, using Vault-injected
certificate material instead of storing real private keys or certificates in
OpenShift Secrets.

It uses public-safe names under `mycompany.org`. It is not a generic TLS theory
document.

## Final Answer

Inject PEM files, not prebuilt KDB files.

The IBM MQ container expects PEM private keys and certificates under the
`/etc/mqm/.../pki` mount points. At startup, MQ converts those PEM files into
runtime CMS key databases under `/run/runmqserver`.

The init container should write:

```text
tls.key
tls.crt
ca.crt
```

The init container should not create:

```text
key.kdb
key.sth
key.rdb
key.crl
```

Those runtime files are generated by the MQ container.

Server-side apply showed mixed output, but the stored QueueManager object did not

## Init Container Design

The init container uses the same IBM MQ image as the `qmgr` container:

```text
icr.io/ibm-messaging/mq:9.4.5.0-r2
```

It runs:

```text
bin/mq_vault_cert_init.sh
```

The script uses only tools available in the MQ image:

```text
curl
openssl
sed
sh/bash
```

It fetches Vault KV v2 secrets using:

```text
X-Vault-Token: <token>
GET /v1/kv/data/mq/TMPQM001/<cert-path>
```

The script validates each leaf cert/key pair by:

```text
openssl x509 -in tls.crt -noout -checkend 3600
openssl x509 -in tls.crt -pubkey -noout
openssl pkey -in tls.key -pubout
```

Then it writes the PEM files into the shared runtime emptyDir mounted into the MQ
container.

## Multus Issue Seen During Lab

One failure during testing was unrelated to MQ TLS.

Pods were stuck in `Init:0/1` because Multus had stale UID / authorization
errors. The symptom was:

```text
expected pod UID "<current>" but got "<old>" from Kube API
Unauthorized
```

The targeted fix was to restart Multus daemon pods on affected nodes:

```text
openshift-multus/multus-abcde on worker-a
openshift-multus/multus-fghij on worker-b
```

After Multus restarted, MQ pods moved to `Running`, and the QueueManagers reached
`Running`.

## Final Lab Position

The proven architecture is:

```text
App PKI:
  Vault -> init container -> /etc/mqm/pki -> MQ generates /run/runmqserver/tls/key.kdb

NativeHA internal TLS:
  Vault -> init container -> /etc/mqm/ha/pki -> MQ generates /run/runmqserver/ha/tls/key.kdb

NHACRR group TLS:
  Vault -> init container -> /etc/mqm/groupha/pki -> merged into /run/runmqserver/ha/tls/key.kdb

OpenShift Secret:
  tmpqm001-admission-shim only, dummy one-day shim for IBM MQ Operator admission/controller behavior
```

The cert/key material that matters at runtime is Vault-injected PEM. The KDBs are
generated inside the pod by IBM MQ startup logic.
